1Introduction & Background

Private tutoring is one of the most significant supplementary education markets in Bangladesh. Millions of students seek home tutors for school, college, and university subjects every year. Despite the scale of the market, the process of finding, verifying, and hiring a tutor remains largely informal — relying on word-of-mouth, social media groups, or local middlemen.

TuitionMedia digitises this ecosystem — bringing students, tutors, and administrators together on a single verified, transparent, and intelligent platform.

For Students

  • Post detailed tuition requirements
  • Browse & filter verified tutors
  • Unlock tutor contact details
  • Track tuition history

For Tutors

  • Build a verified public profile
  • Browse & apply to student posts
  • Manage active tuitions
  • Pay commissions digitally

3Functional Requirements

Authentication & Accounts

FR-01
Phone-based Registration

The system shall allow users to register using a unique phone number and password, assigning a role of Student or Tutor.

Critical
FR-02
OTP Password Reset

The system shall provide a forgot-password flow using SMS OTP sent to the registered phone number.

Critical
FR-03
Role-based Dashboard

Upon login, users shall be redirected to a personalised dashboard based on their role (Student / Tutor / Admin).

Critical
FR-04
Profile Management

Users shall be able to update personal details, upload a profile photo, and upload an ID document. Profile edits reset approval status.

High
FR-05
Profile Approval Gate

Admin must approve a tutor profile before it becomes publicly visible in search results.

Critical

Posts & Tuition Matching

FR-06
Create Tuition Post

Students shall create posts specifying subject, location, budget, schedule, and details. Posts require admin approval before being visible.

Critical
FR-07
Browse Posts (Tutor View)

Tutors shall browse all active, approved tuition posts and apply with a single click.

Critical
FR-08
Browse Tutors (Student View)

Students shall browse all approved tutor profiles, filter by subject and location, and initiate a tuition request.

Critical
FR-09
Tuition Request Workflow

Either party may send a TuitionRequest. The recipient can accept or reject. On acceptance, a Tuition record is automatically created.

Critical

Payments & Transactions

FR-10
Contact Unlock Payment

Students shall pay a platform fee via digital payment gateway to unlock a tutor's full contact details (phone, address).

Critical
FR-11
Commission Payment

Tutors shall pay a 30% commission on first-month salary, either digitally via payment gateway or by manual proof upload.

Critical
FR-12
Payment History

All payment transactions shall be recorded and accessible by the payer and admin with full status and metadata.

High
FR-13
Salary Entry by Admin

Admins shall set the confirmed salary amount for active tuitions to trigger commission calculation.

High

Messaging & Notifications

FR-14
Permission-gated Messaging

Users may message each other only when an accepted TuitionRequest exists between them. Admins can message anyone.

High
FR-15
AJAX Real-time Chat

The messaging interface shall update with new messages every 3 seconds without a full page reload.

High
FR-16
Unread Message Badges

Unread message counts shall appear per contact in the sidebar and as a total in the navbar, refreshed every 8 seconds.

Medium
FR-17
Push Notifications

The system shall send browser push notifications for key events using VAPID Web Push. Users may opt in on first visit.

Medium
FR-18
In-app Notification Panel

A slide-in panel shall show recent notifications with read/unread status, time, and a link to the relevant resource.

High

Administration

FR-19
Profile & Post Approvals

Admins shall approve or reject pending tutor profiles and student posts from a centralised approval panel.

Critical
FR-20
User Management

Admins shall view all users, access individual profiles, and toggle ban status to restrict platform access.

Critical
FR-21
Admin Inbox

Admins shall have a dedicated inbox to message any user, with AJAX conversation loading, send, unread badges, and polling.

High
FR-22
Guru AI Admin Mode

Admins shall access a Guru AI assistant with live platform statistics for operational insights.

Medium

10Sample Usage Scenarios

Scenario 1 — Student Finds and Hires a Tutor

1
RegistrationFatema signs up with phone 01711XXXXXX. System creates a Student-role account.
2
Profile CompleteFatema fills her profile: class 10, school name, Dhaka. profile_approved = True by default for students.
3
Create PostFatema creates a tuition post: "Math & Physics, Class 10, Mirpur, 3000 BDT/month". Post enters pending_approval.
4
Admin ApprovesAdmin reviews and approves the post. Fatema receives a notification. The post is now visible to tutors.
5
Browse TutorsFatema finds Karim (tutor) with approved profile, Math & Physics expertise.
6
Unlock ContactFatema pays a platform fee via digital payment to unlock Karim's phone and address. ContactUnlock record created.
7
Tuition RequestFatema sends a TuitionRequest to Karim. Karim receives a notification and accepts.
8
Tuition BeginsA Tuition record is created. Chat between Fatema and Karim is unlocked. Admin sets salary = 3000 BDT, commission = 900 BDT (30%).
9
Commission PaidKarim pays 900 BDT commission via payment gateway. Commission status changes to 'paid'. Tuition is active.

Scenario 2 — Admin Manages Platform

1
Admin LoginAdmin logs in with admin credentials. Redirected to Overview dashboard with live statistics.
2
Profile ApprovalsAdmin sees 3 pending profiles. Reviews ID documents. Approves 2, rejects 1 (incomplete submission).
3
Post ApprovalsAdmin reviews 2 pending tuition posts. Approves both after verifying they are legitimate.
4
Admin InboxAdmin opens Inbox. Sees "2 unread" badge next to Fatema. Clicks her name — conversation loads via AJAX instantly.
5
Send MessageAdmin types a reply, clicks Send. Message bubble appears instantly without page reload. Unread badge clears.
6
Guru AI AnalyticsAdmin opens Guru AI, asks "Show me this week's stats". Guru queries live DB and responds with user counts, revenue, pending commissions.

12Security Design

ThreatMechanismImplementation
CSRF AttackCSRF TokenDjango CsrfViewMiddleware; all AJAX POST requests include X-CSRFToken header
Unauthorised AccessLogin Guard@login_required decorator + admin_required() wrapper on all protected views
Privilege EscalationRole Enforcementadmin_required() checks request.user.role == 'admin' for every admin view
Password TheftPBKDF2 HashingDjango's built-in password hasher (PBKDF2-SHA256 + random salt)
Session HijackingSecure CookiesSESSION_COOKIE_SECURE and CSRF_COOKIE_SECURE = True in production
Credential LeakEnv VariablesAll API keys loaded via os.getenv(); secrets never appear in source code
SQL InjectionORM QueriesAll DB access via Django ORM with parameterised queries; no raw SQL
XSSAuto-escapingDjango templates auto-escape all user content; escHtml() in JS
Fake ProfilesApproval Gateprofile_approved flag; tutors hidden until manually verified by admin
Banned User AccessBan Flag Checkbanned field on User; login blocked for banned accounts
