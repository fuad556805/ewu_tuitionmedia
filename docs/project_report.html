Section 01

Introduction & Background

Private tutoring is one of the most significant supplementary education markets in Bangladesh. Millions of students seek home tutors every year for school, college, and university subjects. Despite the scale of the market, the process of finding, verifying, and hiring a tutor remains largely informal — relying on word-of-mouth referrals, social media posts, or local middlemen.

These methods are slow, untrustworthy, and unscalable. Parents cannot verify a tutor's credentials, tutors cannot showcase their work, and no financial trail exists for commission or salary agreements. TuitionMedia is designed to solve all of these problems in a single, integrated web platform.

🎯 Project Goals

For Students

  • Post detailed tuition requirements
  • Browse and filter verified tutors
  • Unlock contact details via payment
  • Communicate directly in-platform
  • Track tuition and payment history

For Tutors

  • Build a verified public profile
  • Browse and apply to student posts
  • Receive direct tuition requests
  • Manage active tuitions
  • Pay commissions digitally (bKash/Nagad)

Section 02

Requirements Analysis

Requirements were elicited by analysing the target user group (students, parents, tutors in Bangladesh), studying existing informal processes, and reviewing comparable platforms. The analysis produced the following key findings:

📌 Stakeholder Analysis

StakeholderGoalPain PointPlatform Role
Student / ParentFind a reliable tutor fastCan't verify credentialsBrowse, unlock, request
TutorAcquire students, prove qualityNo structured showcaseProfile, apply, tuition mgmt
AdminMaintain quality & revenueManual, error-prone approvalFull platform control panel
Payment ProviderProcess transactionsLow digital adoptionAPI integration (bKash/Nagad)
AI ProviderAPI usage growth—Guru assistant backbone

📐 System Boundary

The system interacts with the following external entities:

External Inputs
  • User registrations and profile data
  • Tuition post details from students
  • Tuition applications from tutors
  • Payment callbacks from bKash / Nagad
  • AI responses from Gemini / Groq
  • OTP confirmations from SMS provider
External Outputs
  • Verified tutor contact information
  • Payment receipts and history
  • Push / SMS notifications to users
  • Commission confirmation to tutors
  • Approval / rejection status updates
  • Guru AI responses to users
Section 03

Functional Requirements

👤 Authentication & Accounts

FR-01
Phone-based Registration

The system shall allow users to register using a unique phone number and password, assigning a role of Student or Tutor.

Critical
FR-02
OTP Password Reset

The system shall provide a forgot-password flow using SMS OTP sent to the registered phone number.

Critical
FR-03
Role-based Dashboard

Upon login, users shall be redirected to a personalised dashboard based on their role (Student / Tutor / Admin).

Critical
FR-04
Profile Management

Users shall be able to update personal details, upload a profile photo, and upload an ID document. Edits reset approval status.

High
FR-05
Profile Approval Gate

Admin must approve a tutor profile before it becomes publicly visible in search results.

Critical

📋 Posts & Matching

FR-06
Create Tuition Post

Students shall be able to create posts specifying subject, location, budget, schedule, and details. Posts require admin approval before being visible.

Critical
FR-07
Browse Posts (Tutor View)

Tutors shall be able to browse all active, approved tuition posts and apply to them with a single click.

Critical
FR-08
Browse Tutors (Student View)

Students shall be able to browse all approved tutor profiles, filter by subject and location, and initiate a tuition request.

Critical
FR-09
Tuition Request Workflow

Either party may send a TuitionRequest. The recipient can accept or reject. On acceptance, a Tuition record is automatically created.

Critical

💰 Payments

FR-10
Contact Unlock Payment

Students shall pay a platform fee (via bKash or Nagad) to unlock a tutor's full contact details (phone, address).

Critical
FR-11
Commission Payment

Tutors shall pay a 30% commission on first-month salary. Payment can be made digitally or via manual proof upload.

Critical
FR-12
Payment History

All payment transactions shall be recorded and accessible by the payer and admin with full status and metadata.

High
FR-13
Salary Entry by Admin

Admins shall be able to set the confirmed salary amount for active tuitions to trigger commission calculation.

High

💬 Messaging & Notifications

FR-14
Permission-gated Messaging

Users may only message each other if an accepted TuitionRequest exists between them. Admins can message anyone.

High
FR-15
AJAX Real-time Chat

The messaging interface shall update with new messages every 3 seconds without requiring a page reload.

High
FR-16
Unread Message Badges

Unread message counts shall be displayed per contact in the sidebar and as a total in the navbar, refreshed every 8 seconds.

Medium
FR-17
Push Notifications

The system shall send browser push notifications for key events using VAPID Web Push. Users may opt in on first visit.

Medium
FR-18
In-app Notification Panel

A slide-in panel shall show the user's recent notifications with read/unread status, time, and a link to the relevant resource.

High

🛡️ Administration

FR-19
Profile & Post Approvals

Admins shall be able to approve or reject pending tutor profiles and student posts from a centralised approval panel.

Critical
FR-20
User Management

Admins shall be able to view all users, access individual profiles, and toggle ban status to restrict platform access.

Critical
FR-21
Admin Inbox

Admins shall have a dedicated inbox to message any user directly, with AJAX message loading, send, unread badges, and real-time polling.

High
FR-22
Guru AI Admin Mode

Admins shall have access to a Guru AI assistant with live platform statistics (users, commissions, pending tasks) for operational insights.

Medium
Section 04

Non-Functional Requirements

⚡ Performance

NFR-01
Page Load Time

All template pages shall render within 1.5 seconds under normal load. Static files are served by WhiteNoise with compression enabled.

High
NFR-02
API Response Time

REST API endpoints shall return responses within 300 ms for standard CRUD operations, excluding payment gateway callbacks.

High
NFR-03
Efficient Query Design

The ORM shall use select_related() and prefetch_related() for all foreign key traversals. Unread counts shall use aggregated DB queries rather than in-Python loops.

High

🔒 Security

NFR-04
CSRF Protection

All state-changing POST requests (forms and AJAX) shall include a valid CSRF token validated by Django middleware.

Critical
NFR-05
Authentication Enforcement

All views except landing, login, and signup shall require authentication. Admin views require role='admin' check.

Critical
NFR-06
Secure Cookies in Production

SESSION_COOKIE_SECURE and CSRF_COOKIE_SECURE shall be True in non-debug environments.

Critical
NFR-07
Password Hashing

All passwords shall be stored as PBKDF2 hashes using Django's built-in password hasher. Plaintext passwords shall never be persisted.

Critical
NFR-08
Secrets Management

API keys, payment credentials, and JWT secrets shall be loaded exclusively from environment variables. No credentials shall appear in source code.

Critical

📱 Usability

NFR-09
Responsive Design

All pages shall be usable on desktop, tablet, and mobile (minimum 320px width) without horizontal scroll.

High
NFR-10
Theme Switching

The system shall support three visual themes (Dark Purple, Ocean Blue, Sunset Gold) persisted per user account.

Medium
NFR-11
No Reload Interactions

Message sending, loading conversations, and unread count polling shall not require full page reloads (AJAX/fetch API).

High

🔧 Maintainability

NFR-12
Modular App Structure

Code shall be organised into independent Django apps (accounts, posts, tuitions, payments, chat, guru, admin_panel) with clear domain ownership.

High
NFR-13
Database Migrations

All schema changes shall be managed through Django's migration system. No raw DDL shall be executed manually against production.

Critical
NFR-14
Configurable SMS Backend

The SMS provider shall be swappable via a single environment variable (console, twilio, bulksmsbd, sslwireless) without code changes.

Medium

📈 Scalability & Reliability

NFR-15
Gunicorn Production Server

The production deployment shall use Gunicorn as the WSGI server with configurable worker count to handle concurrent requests.

Critical
NFR-16
AI Fallback Strategy

The Guru AI module shall support a dual-provider strategy (Gemini primary, Groq fallback) to maintain availability during API outages.

High

Unauthorised AccessAuthentication Guard@login_required decorator + admin_required wrapper on all protected views
Privilege EscalationRole Enforcementadmin_required() checks request.user.role == 'admin' for all admin views
Password TheftPBKDF2 HashingDjango's built-in password hasher (PBKDF2-SHA256) with salt

Credential LeakEnvironment VariablesAll API keys loaded via os.getenv(); .env excluded from VCS
SQL InjectionORM ParameterizationAll DB access via Django ORM with parameterised queries
XSSTemplate Auto-escapingDjango templates auto-escape all user content; escHtml() in JS for dynamic content
Fake ProfilesAdmin Approval Gateprofile_approved flag; tutors invisible until manually verified by admin

  • Verified profiles eliminate fake tutors
  • Structured posts accelerate discovery
  • Digital payments create audit trails
  • Permissioned chat ensures privacy
